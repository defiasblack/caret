# Support

## Usage questions

Start with the [README](README.md), built-in `F1` help, and the documentation in [`docs/`](docs/).

For questions about using Caret, installation, keybindings, plugins, or configuration, open a GitHub Discussion when discussions are available. Include your Caret version, operating system, terminal emulator, and shell so others can help efficiently.

## Bug reports

Use the repository's bug-report form for reproducible defects. Before opening a report:

1. Search existing issues.
2. Try the latest Caret release when practical.
3. Reduce the problem to the smallest reproducible example.
4. Remove credentials, tokens, private paths, and personal information.

## Feature requests

Use the feature-request form and explain the user problem before describing a proposed implementation. Early design ideas are better suited to a Discussion when available.

## Security issues

Do not disclose vulnerabilities publicly. Follow [SECURITY.md](SECURITY.md) and use GitHub private vulnerability reporting.

## Response expectations

Caret is maintained on a best-effort basis. Clear, reproducible reports are easier to investigate, but response and resolution times cannot be guaranteed.
